Cascade Component: 3 Approaches

How should the side-panel expansion and multi-select be architected internally?

A

Single flat state machine (Recommended)

State = activePathIndices: number[] tracking which column is open at each depth, plus value. Each column renders from the options tree by following the active path. Multi-select uses a flat Set<string> of selected leaf values.

Pros

  • Simple, predictable state
  • Easy to reason about column rendering
  • Search just filters the root level, then resets path

Cons

  • Parent "select all" requires walking the subtree to collect leaves
B

Recursive column components

Each column is a self-contained <CascadeColumn> component that receives its slice of options and renders the next column recursively when an item is active.

Pros

  • Natural recursive structure mirrors the data
  • Each column is independently testable

Cons

  • State lifted to root anyway (Dash needs it)
  • More component boilerplate for the same result
  • Harder to animate column transitions
C

Pre-computed index at build time

On mount, flatten the entire options tree into a lookup map valueToNode and a valueToPath map. All operations (select, select-all, search) work against the index.

Pros

  • O(1) lookups for select-all and path resolution
  • Cleanest search implementation

Cons

  • Extra complexity on mount/options-change
  • Overkill for typical small-to-medium option trees
