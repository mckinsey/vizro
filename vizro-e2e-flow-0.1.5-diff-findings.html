C. Substantive content additions BRING UPSTREAM

C1. New AG Grid reference (references/examples/example_ag_grid.py, 165 lines)

Canonical patterns:

  • Drop-in via dash_ag_grid factory — column defs, cell styling, sizing, pagination.
  • Custom via @capture("ag_grid") when you need full control.
  • Gotchas list: date formatting, pagination defaults, sticky-header limits, when client-side callbacks fail.
  • Explicit guidance: if AG Grid fails Playwright twice in a row, substitute a simpler chart.

C2. New Playwright smoke-test reference (references/examples/example_playwright_check.py, 140 lines)

Collects all pageerror, console.error, and HTTP ≥ 500 events across every page in a single walk, deduplicates by fingerprint, prints one combined report. Includes per-page filter-action map so filter callbacks fire too.

C3. vizro-dashboard-build Step 2 (Testing) rewrite

Massively expanded from "click through, check console" to a concrete batch-error workflow:

  • playwright install --with-deps chromium with skip-if-cached guard. Cold install needs --with-deps on slim images (libnss3, libatk-bridge, libxkbcommon) or Chromium fails to launch.
  • "Discover ALL callback errors in one walk, fix in one batch" — explicit framing that each restart cycle is ~50 s, so 5 fix-restart loops waste ~4 min.
  • Toggle every filter once (DatePicker, multi-select, dropdown) so filter callbacks fire — initial-render walks miss filter-callback bugs.
  • grep -A 30 "Exception on /_dash-update-component" /tmp/dash.log for server-side tracebacks. Playwright sees the symptom, dash.log has the cause.
  • Group failures by exception class + responsible chart, plan one fix per group, batch the edits with MultiEdit, restart once, re-run.
  • Escape hatch: if the same component fails twice in a row even after distinct fix attempts, stop iterating on it. Substitute the next-simplest visual (table → bar chart; AG Grid → KPI cards row; complex multi-series → faceted small-multiples). Design spec is authoritative for what, not how.
